  <files>src/SectorManager.js</files>
  <action>
Add an `assignResourceProfile(x, y)` method to SectorManager that:

1. Calculates Euclidean distance from origin: `Math.sqrt(x * x + y * y)`
2. Uses distance-based weight tables for profile selection:
   - Distance < 5: weights [60, 20, 10, 8, 2] for [balanced, mineral-rich, data-rich, artifact-rich, probethium-rich]
   - Distance 5-10: weights [45, 25, 15, 10, 5]
   - Distance 10+: weights [30, 25, 20, 15, 10]
3. Performs weighted random selection (same pattern as existing sector type selection at lines 211-222)
4. Returns object: `{ type: 'mineral-rich', spawnRateMultiplier: 1.5, assignedDistance: distance }`

Spawn rate multiplier values:
- balanced: 1.0
- mineral-rich: 1.5
- data-rich: 1.5
- artifact-rich: 1.5
- probethium-rich: 0.8

Integrate into `initializeSector()` (around line 228-238):
- After creating the sector object, add: `sector.resourceProfile = this.assignResourceProfile(x, y);`
- This means ALL new sectors get a profile on creation

Integrate into `discoverSector()` (around line 116-149):
- When discovering an EXISTING sector that was initialized but not explored (the `else` branch at line 117), check if it has a `resourceProfile`. If not, assign one: `if (!sector.resourceProfile) sector.resourceProfile = this.assignResourceProfile(x, y);`
- This handles sectors that may have been created before this feature existed

Profiles are INDEPENDENT from sector types per CONTEXT.md. A Resource-Rich sector (Ore Vein signals) can be data-rich profile (mining produces data). Do NOT couple profile assignment to sector type.

Save compatibility: No changes to SaveManager needed. Sectors are serialized via spread operator (`...sector`) in `serializeSectors()` (line 254), so resourceProfile saves automatically. On load, old sectors without profiles get defensive fallback handling in consumers (MiningManager, ProbeManager).
